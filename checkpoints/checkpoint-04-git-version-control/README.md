# Checkpoint 04 — Git and Version Control

## Goal

Understand Git as version control and GitHub as one place where Git repositories can be hosted.

## Instruction

Do these tasks in order:

1. Initialize Git in your project folder.
2. Make at least five meaningful commits.
3. Intentionally break something small.
4. Commit the broken state with a clear message.
5. Fix the problem.
6. Commit the fix.
7. Push the repository to GitHub.
8. Write the difference between Git and GitHub.
9. Record one Git error or confusion you faced.

## Example

Example only. Do not copy these exact commit messages unless they match your work.

```text
git init
git add .
git commit -m "Add first signal simulator"
git commit -m "Save sample readings to CSV"
git commit -m "Document signal choices"
git commit -m "Break CSV output intentionally"
git commit -m "Fix CSV output bug"
```

## Submit

Send these items to the mentor:

1. Screenshot of `git log`.
2. Repository link.
3. Explanation of commit, branch, and remote.
4. Explanation of Git versus GitHub.
5. One Git error or confusion and how you searched for it.

## Done when

This checkpoint is complete when:

- your project is tracked with Git
- your commits show meaningful progress
- your repository is available on GitHub
- you can explain Git without saying only "GitHub"

## Mentor review

The mentor may ask:

- Can Git work without GitHub?
- What is a commit?
- What is a remote?
- What is a branch?
- Why should commits be meaningful?
